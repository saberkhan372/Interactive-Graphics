Unit 5 · Lesson 5.2

GBVM and Sprites

GB Studio works because it gives students a safer layer above the hardware. GBVM interprets scripts and changes game state, while sprites turn small pixel sheets into actors, animation, and readable movement.

Source decks: IG 3.3 - Game Boy Studio.pptx · IG 3.5 - Sprites.pptx

What the class did

The lesson introduced GBVM as the virtual machine that makes GB Studio possible. Instead of asking students to write everything in C or assembly, GBVM interprets bytecode and calls internal functions that change the global state of the game. The deck named control over execution, safety, and concurrent script execution as reasons this layer matters.

The sprite portion focused on asset constraints: simple sprites at 16 by 16 pixels per frame, static sprites, animated sprites, actors with directional frames, and animated actors. The class also named Game Boy visual techniques such as color cycling, palette shifting, sprite stitching, and sprite stacking.

Code shown

// GB Studio sprite dimensions from the lesson
simple sprite:  16 x 16 per frame
actor:          48 x 16 for three directions
animated actor: 96 x 16 for six frames

Key idea

A sprite is not just an image. It is an asset with expected dimensions, frames, directions, animation states, and palette limits.

Try it

Use the sprite editor's mirror control to build a character that can face left and right. GB Studio can generate some directions, but the original frame still has to read clearly.
