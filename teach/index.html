Teacher's Guide

This guide is for running Interactive Graphics as a real course, not just browsing it as an archive. The sequence works best when students keep a notebook, write small programs often, and treat every constraint as a design material.

The goal is not to cover every graphics topic. The goal is to make students feel the historical and mathematical layers under a rendered image: hardware limits, coordinate systems, pixels, transformations, projection, light, and finally tools that hide those layers again.

Before Day One

Set up the class so notebooks and code sit beside each other. Students should be able to sketch a timeline, write a BASIC loop, port it to p5.js, and explain what changed between the two systems.
  • Give every student a notebook. Warm-ups, design sketches, timelines, and research notes belong there before they become code.
  • Bookmark an Apple II BASIC emulator and test HGR, GR, HPLOT, HLIN, VLIN, and simple loops before assigning the first animation.
  • Use the local tools on this site for fast demos, then send students back to their own sketches and p5.js projects.
  • Keep one public portfolio destination in view from the beginning. GitHub Pages works because it turns finished work into something students can actually visit.
  • Decide in advance which assignments allow outside code references and which do not. The AI policy page should be part of that opening conversation.

Pacing Guide

The course can run as a tight semester or as a roomier half-year sequence. If time collapses, protect the historical opening, one BASIC project, one no-library 3D pipeline experience, and one final public artifact.

Unit 0

History and Constraint

4-7 meetings

Start with personal computing history, then move quickly into pixel algorithms. Do not let the history sit as lecture only; students should make a constraint visible by the end of the unit.

Unit 1

BASIC and Fractals

8-12 meetings

This is where time expands. BASIC syntax, emulator friction, research, animation, and presentations all need room. Keep project workdays explicit.

Unit 2

2D to 3D

4-6 meetings

Use the BASIC-to-p5.js port as the bridge. Students should notice that the idea can survive while the language, coordinate system, and draw loop change.

Unit 3

Pipeline and Wireframe

6-9 meetings

The important move is from object coordinates to screen coordinates. Slow down when students confuse moving the object with moving the camera.

Unit 4

3D Scenes and WebGL

10-14 meetings

Build scenes in layers: primitives, transforms, light, material, camera, texture, custom geometry. A polished scene takes more debugging time than the slides suggest.

Unit 5

Game Dev Bridge

4-7 meetings

GB Studio works best as a constrained bridge, not a full game-design course. Keep the focus on palettes, sprites, triggers, and how engine tools change authorship.

What Actually Slips

  • Cancelled and subbed classes are normal. Keep one short, tool-based lesson ready that does not require a new assignment launch.
  • Notebook research takes longer than expected, especially when students are choosing a historical topic for animation.
  • Project presentations rarely fit into one day if students show code, explain research, and screen a finished animation.
  • WebGL debugging expands quickly. Leave buffer days after introducing camera, texture, and custom geometry.
  • School calendar interruptions matter: conferences, spring break, intersession, and senior schedules can remove continuity right when students need it most.

Warm-Up Bank

These are notebook questions, not exit tickets. They give students a way to rehearse the idea before the code makes the idea more brittle.

Opening

Why are you taking this class? What does interactive graphics mean to you?

History

What developments made animation and filming possible, and how do they connect to computer graphics?

Apple II

What feels different about writing code for the Apple II? What still feels familiar?

Research

What topic from graphics, film, animation, or computer science are you choosing, and why?

Design

Describe the first frame, a middle frame, and the final frame of your BASIC animation.

Porting

How is p5.js different from BASIC? What idea from your BASIC sketch can stay the same?

3D

What has to be added to a 2D shape before it can become a 3D object?

Pipeline

Where does the object exist before it becomes pixels on the screen?

Camera

What changes when you move the camera instead of moving the objects?

Light

How does a material tell light what kind of surface it is?

Texture

What information does a texture add that geometry alone cannot carry?

Reflection

Which constraint made your work harder, and which constraint made it more interesting?

Per-Unit Teaching Notes

Unit 0

Anchor the course in access, hardware, and visual culture. The historical arc should end with students making a pixel rule, not just naming old machines.

  • Use: Pixel Algorithm Playground
  • Watch for: history becoming trivia instead of design evidence.

Unit 1

Let BASIC feel awkward. The awkwardness is part of the lesson because it exposes memory, loops, coordinates, and the screen as a real machine.

  • Use: BASIC/p5.js Twin Panels
  • Watch for: students copying code before they can narrate it.

Unit 2

Make porting intellectual, not clerical. Ask students to identify which parts of the idea are language-specific and which parts are structural.

  • Use: Flatten/Extrude
  • Watch for: treating p5.js as a prettier BASIC instead of a different environment.

Unit 3

The pipeline is the hinge of the course. Students should be able to point to object space, world space, camera space, and screen space in their own words.

  • Use: Pipeline Step-Through
  • Watch for: matrix language hiding the simpler question of where a point is.

Unit 4

WebGL can become a bag of features. Keep asking what each feature changes in the image and what calculation it hides.

  • Use: Camera & Perspective Lab
  • Watch for: beautiful scenes with no explanation of camera, light, or geometry.

Unit 5

GB Studio is strongest when it returns the course to constraint: tile size, palette, sprite state, scene triggers, and hardware-feeling limits.

  • Use: Game Boy Sprite Editor
  • Watch for: engine menus replacing design decisions.

Assessment Rhythm

  • Grade major work as projects with visible requirements, comments, and a short reflection, not only as code correctness.
  • Ask students to show process: notes, sketches, research sources, code decisions, and what changed after critique.
  • Keep outside help visible. Students should cite references, tutorials, peers, and approved AI help in the same spirit: name what helped and what they changed.
  • End with a portfolio so the course has a public shape beyond the gradebook.
