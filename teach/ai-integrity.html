Course policy

How We Use AI in This Course

The central question of Interactive Graphics is not just how to make an image. It is what you understand about the system that made the image possible. Generative AI makes that question sharper: what do you keep when the machine can write code for you?

In this course, the default rule is simple: do not use generative AI for any part of an assignment unless the teacher explicitly allows it. That includes brainstorming, research summaries, writing, debugging, code generation, and revision. When AI use is allowed, it has to be visible, limited, and explained.

Your idea stays yours

You choose the topic, visual argument, constraints, references, and final decisions. A tool can suggest; it cannot replace your authorship.

Your code stays legible to you

If you cannot explain what a line does, you should not submit it as your work. Understanding matters more than speed.

Your help stays documented

Outside assistance belongs in the open: tutorials, classmates, teachers, documentation, and approved AI tools all need credit.

Allowed Only When Assigned

Sometimes the teacher may ask you to use an AI system as an object of study. You might compare an AI explanation to your own notes, ask it for a second version of a small function, or critique a generated description of BASIC and p5.js. In those cases, the AI output is not the final answer. It is material you are responsible for reading, testing, revising, and questioning.

What Must Be Documented

  • The tool you used and what you asked it to do.
  • Which part of the output you used, rejected, or changed.
  • How you tested that the code or explanation was correct.
  • What you now understand that you did not understand before.

What Is Not Acceptable

  • Submitting AI-generated code, text, images, or research as if you made it yourself.
  • Using AI to bypass the hard part of an assignment: choosing, sketching, coding, debugging, or explaining.
  • Replacing your notebook process with generated summaries or plans.
  • Asking AI for a finished project and then changing surface details.

A Note from the Archive

The 2023-2024 lesson planner includes a January 2024 moment where a generated comparison between Applesoft BASIC and p5.js was used in lesson prep. That is useful to keep in the public history of the course because it shows the exact problem students now need to learn how to handle: AI can produce a plausible paragraph, but a plausible paragraph is not the same thing as understanding the two systems.

So the classroom move is not "AI is magic" or "AI is forbidden because it exists." The move is stricter and more interesting: read the output as a claim, test it against code, revise it in your own language, and name the work you did.

Submission Note Template

When AI use is explicitly allowed, add a short note like this to your project document or code comments:

AI use note: I used [tool] to [limited task]. I used [specific part] after checking it against [test, documentation, or code]. I changed [specific change]. The final design, code decisions, and explanation are mine.

Teacher Check

  • Before assigning: say whether AI is allowed, not allowed, or required as an object of critique.
  • During work time: ask students to explain the code path, not just show that it runs.
  • During critique: ask what changed because of constraint, not only what looks polished.
  • After submission: read the process note alongside the artifact.
